R-Setup auf Ubuntu (Linux)

Wenn install.packages() mit „fatal error: foo.h: No such file or directory" abbricht, fehlen System-Header. Hier steht die Lösung — und wie du sie selbst findest.

Schnelllösung — fehlende Header

Für tidyverse auf Ubuntu (getestet Ubuntu 26.04, R 4.5): 

sudo apt-get update
sudo apt-get install -y \
  libcurl4-openssl-dev \
  libfontconfig1-dev \
  libfreetype-dev \
  libfribidi-dev \
  libharfbuzz-dev \
  libpng-dev libjpeg-dev libtiff-dev

Danach in R einfach erneut:

install.packages("tidyverse")
Falle: Paketnamen ändern sich

Die R-Fehlermeldung ([ANTICONF]) schlägt oft ältere Namen vor. Auf Ubuntu 24.04+/26.04 existiert libfreetype6-dev nicht mehr — richtig ist libfreetype-dev (ohne „6"). Immer prüfen mit:

apt-cache policy <paketname>   # Candidate: (none) = existiert nicht
apt-cache search freetype        # nach dem echten Namen suchen

Warum passiert das überhaupt?

R-Pakete mit C/C++/Fortran-Code müssen kompiliert werden. Auf Windows und macOS liefert CRAN fertige Binaries — ein Klick. Auf Linux kompiliert R aus dem Quelltext, weil es für jede Distribution/Version eigene Binaries bräuchte. Zum Kompilieren braucht R die Entwickler-Header (die -dev-Pakete) der Systembibliotheken, gegen die gelinkt wird.

PaketLiefert Header für …R-Paket, das es braucht
libcurl4-openssl-devlibcurl (HTTP)curl → httr, rvest
libfontconfig1-devfontconfigsystemfonts
libfreetype-devfreetype2systemfonts, ragg
libfribidi-dev + libharfbuzz-devBidi-/Text-Shapingtextshaping → ragg
libpng-dev / libjpeg-dev / libtiff-devBildformateragg, Grafik-Devices

Die Fehlermeldung liest die Lösung schon vor

Jeder configure-Abbruch enthält einen [ANTICONF]-Block, der genau sagt, welches Systempaket fehlt — inklusive der Distro-Zeile (deb: … für Debian/Ubuntu, rpm: … für Fedora). Gewöhne dir (und deine Schüler) an, diesen Block zu lesen, statt die Wand aus Text zu überfliegen:

--------------------------- [ANTICONF] --------------------------------
Configuration failed because libcurl was not found. Try installing:
 * deb: libcurl4-openssl-dev (Debian, Ubuntu, etc)
 * rpm: libcurl-devel (Fedora, CentOS, RHEL)
----------------------------------------------------------------------

Nur eine Einschränkung: der vorgeschlagene Name kann veraltet sein (siehe libfreetype6-dev-Falle oben) — kurz mit apt-cache policy gegenchecken.

Lösmuster — immer dasselbe Rezept

  1. Fehlermeldung lesen, [ANTICONF]-Block suchen.
  2. Vorgeschlagenes -dev-Paket mit apt-cache policy prüfen; nötigenfalls per apt-cache search echten Namen finden.
  3. sudo apt-get install -y <paket>
  4. In R: install.packages("<fehlendes R-Paket>") neu ausführen — es zieht automatisch alle dependents nach.

Alternative: fertige Binaries

Wer gar nicht kompilieren will, konfiguriert einmalig den Posit Public Package Manager als Binary-Repo. Für Ubuntu 26.04 liefert PPM aktuell noch keine Binaries — deshalb ist hier der Klassiker-Weg (Header installieren) die robuste Lösung. Auf älteren Ubuntus (22.04/24.04) lohnt sich PPM.
